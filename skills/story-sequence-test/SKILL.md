# 一句话序列故事检验

这是最轻量、最快速的故事验证工具。核心思想极其简单：**如果你不能用一句话概括一个故事序列，说明这个序列还不成立。**

## 核心原理

一个成立的故事序列必须具备：
- 一个清晰的主角
- 一个明确的目标
- 一个具体的障碍
- 一个结果（成功或失败，但必须有变化）

如果这些要素模糊不清，一句话就说不出来——这恰恰说明序列本身有问题。

## 检验公式

```
[谁] 为了 [什么目标]，面对 [什么障碍]，做了 [什么行动]，结果 [什么变化]。
```

**合格的一句话序列故事**：
- "一个失业的父亲为了保住女儿的抚养权，面对没有收入和前妻的律师团队，假扮女性去做保姆，结果学会了真正理解孩子。"
- "一个胆小的程序员为了拯救被黑客绑架的AI伙伴，面对自己不会社交的障碍，不得不在现实中组建团队，结果发现真正的连接不需要代码。"

**不合格的一句话序列故事**：
- "一个人经历了很多事情，最终成长了。"（缺目标、缺障碍、缺行动）
- "一个关于友情的故事。"（只有主题，没有故事）
- "主角和反派打了一架。"（有行动但没有目标和变化）

## 操作流程

### Step 1：写出你的一句话

用检验公式写出你故事（或故事序列）的一句话概括。

### Step 2：逐要素检查

| 要素 | 检查问题 | 常见问题 |
|------|----------|----------|
| 谁 | 主角是谁？是否具体？ | 主角太模糊（"一个人"）或太多（"一群人"） |
| 什么目标 | 主角想要什么？是否明确？ | 目标太抽象（"想要幸福"）或太容易（"想喝杯水"） |
| 什么障碍 | 什么阻止了主角？是否有力度？ | 障碍太弱（主角轻易克服）或太强（主角毫无机会） |
| 什么行动 | 主角做了什么来克服障碍？ | 行动与目标不匹配，或主角被动等待 |
| 什么变化 | 故事结束时什么不同了？ | 没有变化，或变化与行动无关 |

### Step 3：诊断问题

根据检查结果，定位问题所在：

```
## 一句话序列检验报告

### 一句话概括
"[谁] 为了 [什么目标]，面对 [什么障碍]，做了 [什么行动]，结果 [什么变化]。"

### 要素检查
| 要素 | 状态 | 问题 | 修复建议 |
|------|------|------|----------|
| 谁 | ✅/❌ | | |
| 什么目标 | ✅/❌ | | |
| 什么障碍 | ✅/❌ | | |
| 什么行动 | ✅/❌ | | |
| 什么变化 | ✅/❌ | | |

### 修改后的一句话
"[修改后的概括]"
```

## 进阶用法

### 序列嵌套检验

长篇故事由多个序列组成。对每个序列分别做一句话检验：

```
整体：[一句话]
├── 序列1：[一句话]
├── 序列2：[一句话]
├── 序列3：[一句话]
└── 序列4：[一句话]
```

**检验规则**：
- 每个序列的一句话必须成立
- 序列之间必须有因果关系（序列1的结果 → 序列2的起因）
- 所有序列合在一起必须等于整体的一句话

### 对比检验

当你有两个故事构思拿不定主意时，分别写一句话，然后对比：

| 维度 | 构思A | 构思B |
|------|-------|-------|
| 主角独特性 | | |
| 目标吸引力 | | |
| 障碍力度 | | |
| 行动戏剧性 | | |
| 变化深度 | | |

哪个在更多维度上更强，就选哪个。

### 电梯演讲检验

一句话序列故事就是你的"电梯演讲"——如果你在电梯里遇到投资人/编辑，能在30秒内说完的故事。如果说不清，不是你口才不好，是故事本身还不够清晰。

## 与其他技能的配合

- 通过检验后，用 **story-three-pillars**（三支柱与八要素）展开详细设计
- 如果一句话写不出来，用 **story-simplify-classify-program**（简化分类程序法）梳理思路
- 如果序列之间缺乏因果，用 **story-roadmap**（路标事件法）重新规划结构
