# PRD：历史故事创作技能（history-story）

## 1. 产品概述

### 1.1 产品定位
一个基于 D.E.S.I.R.E. Matrix 心理分析框架的历史故事创作系统，将"历史上的今天"事件转化为引人入胜的故事，让读者在沉浸式阅读中自然领悟历史智慧。

### 1.2 核心价值主张
- **不直接说教**：让故事本身成为最好的老师，读者自行领悟而非被灌输
- **专属文风定制**：每个历史事件都有独特的心理密码，据此定制专属文风
- **团队协作流水线**：9个专业角色分工协作，确保每个环节的专业性
- **上下文安全**：严格的单事件隔离机制，避免上下文爆炸和污染

### 1.3 目标用户
- 历史爱好者：希望以故事形式了解历史
- 内容创作者：需要历史题材的创作灵感和素材
- 教育工作者：希望用故事化方式讲授历史

## 2. 功能需求

### 2.1 核心功能：历史故事创作

#### F1 历史事件获取
- 从多个数据源并行获取"历史上的今天"事件
- 支持中文和英文数据源，确保中外均衡
- 数据源：Day in History API（世界史）、无铭便民API（中国史）、WebSearch（补充）
- 输出：`events.md`（事件概览列表）

#### F2 深度史实调研
- 对全部事件并行调研，不做筛选
- 调研内容：时间、地点、人物、前因后果、争议解读
- 每个子代理只处理一个事件
- 输出：`research.md`（结构化调研报告）

#### F3 D.E.S.I.R.E. Matrix 分析
- 对全部事件并行进行心理分析
- **事件 Matrix**：分析"这个事件为什么能流传千年"
- **人物 Matrix**：对关键历史人物进行独立心理画像
- 六维度：D-Drive、E-Experience、S-Shortcut、I-Identity、R-Risk、E²-Emotion
- 输出：`matrix.md` + `characters/{人物名}.md`

#### F4 事件评估与选题
- **评估**：对全部事件并行输出紧凑评估卡片（~200字/张）
- **选题**：主控代理读取评估卡片，选出3-5个事件
- 选题标准：综合评分、中外均衡、主题多样性
- 输出：`eval-card.md` + `selection.md`

#### F5 创作画像与要求
- 对入选事件并行输出
- **三合一创作画像**：
  - [MATRIX]：精炼的心理分析
  - [WritingStyle]：基于 Matrix 二次推导的专属文风
  - [剧情建议]：基于 Matrix 的前瞻性剧情指导
- **创作要求**：核心视角、情感定位、必须呈现的史实、认知偏误提醒、禁忌事项、字数建议
- 输出：`profile.md` + `pitch.md`

#### F6 人物设定与关系
- 基于 Matrix 分析创建人物设定卡片
- 创建人物关系图谱
- 人物超过5个时，只为主要人物（3-5人）创建完整设定
- 输出：`characters/{人物名}.md`（更新）+ `relationships.md`

#### F7 故事大纲
- 整合所有素材设计故事大纲
- 包含：场景序列、叙事视角、情感弧线、史实注记预设
- 输出：`outline.md`

#### F8 故事创作
- Storyteller 读取大纲+创作画像+创作要求+人物设定后创作
- 严格遵循专属文风
- 故事末尾附史实注记（✅史实 / 🎭合理想象 / ❓争议）
- 输出：`story.md`

#### F9 史实审核
- 对照 research.md 审核故事的史实准确性
- 审核叙事质量
- 输出：`review.md`

### 2.2 约束条件

#### C1 上下文控制
- **核心规则**：每个子代理只处理一个事件，只读取该事件的文件，绝不跨事件读取
- 唯一例外：Selector 步骤读取所有 eval-card.md（每张~200字）
- 目的：避免上下文爆炸、避免上下文污染、支持并行处理

#### C2 史实底线
- 核心史实（时间、地点、人物、关键事件）不可更改
- 可合理想象心理活动、对话和场景细节
- 可虚构次要人物和情节
- 对有争议的史实，选择最主流学术观点
- 每个故事必须附史实注记

#### C3 不说教原则
- 启示融入故事结构，不直接写出
- 用具体细节代替抽象概括
- 用人物行动代替作者评论
- 用场景描写代替背景介绍

#### C4 中外均衡
- 事件选择需覆盖中国史和世界史
- 避免全是同一地区/时代的事件
- 避免全是同一主题（如全是战争）

## 3. 技术架构

### 3.1 团队架构（9个角色）

| 角色 | 职责 | 处理范围 | 执行方式 |
|------|------|----------|----------|
| Researcher 史料研究员 | 获取事件、深度调研 | 单事件 | 子代理 |
| Analyst 深度分析师 | D.E.S.I.R.E. Matrix 分析 | 单事件 | 子代理 |
| Evaluator 事件评估员 | 轻量级评估，输出评估卡片 | 单事件 | 子代理 |
| Selector 选题决策 | 读取评估卡片，选出3-5个 | 仅读eval-card | 主控代理 |
| PitchEditor 选题编辑 | 三合一创作画像 + 创作要求 | 单事件（仅入选） | 子代理 |
| CharacterDesigner 人物设计师 | 人物设定、关系图谱 | 单事件（仅入选） | 子代理 |
| Architect 故事架构师 | 故事大纲 | 单事件（仅入选） | 子代理 |
| Storyteller 故事讲述者 | 最终故事创作 | 单事件（仅入选） | 子代理 |
| Editor 史实审核员 | 史实审核 | 单事件（仅入选） | 子代理 |

### 3.2 工作流程（8个Phase）

```
Phase 1: 获取历史事件 ──── Researcher（并行数据源）
    ↓
Phase 2: 深度史实调研 ──── Researcher（并行全部事件）
    ↓
Phase 3: Matrix 分析 ───── Analyst（并行全部事件）
    ↓
Phase 4a: 事件评估 ─────── Evaluator（并行全部事件）
    ↓
Phase 4b: 选题决策 ─────── Selector（主控代理，仅读eval-card）
    ↓
Phase 4c: 创作画像 ─────── PitchEditor（并行入选事件）
    ↓
Phase 5: 人物设定 ──────── CharacterDesigner（并行入选事件）
    ↓
Phase 6: 故事大纲 ──────── Architect（并行入选事件）
    ↓
Phase 7: 故事创作 ──────── Storyteller（并行入选事件）
    ↓
Phase 8: 史实审核 ──────── Editor（并行入选事件）
```

### 3.3 并行策略

| Phase | 并行粒度 | 说明 |
|-------|----------|------|
| Phase 1 | 数据源级 | 多个API并行请求 |
| Phase 2 | 事件级 | 每个事件1个子代理 |
| Phase 3 | 事件级 | 每个事件1个子代理 |
| Phase 4a | 事件级 | 每个事件1个子代理 |
| Phase 4b | 无 | 主控代理直接执行 |
| Phase 4c | 事件级 | 每个入选事件1个子代理 |
| Phase 5-8 | 事件级 | 每个入选事件1个子代理 |

### 3.4 目录结构

```
./dayinhistory/                    # 原始数据层
  {MM}/{DD}/
    events.md                      # 事件概览
    {event-slug}/
      research.md                  # 调研结果

./history/                         # 创作层
  {MM}/{DD}/
    selection.md                   # 选题结果
    {event-slug}/
      matrix.md                    # Matrix 分析
      eval-card.md                 # 评估卡片
      profile.md                   # 三合一创作画像
      pitch.md                     # 创作要求
      characters/{人物名}.md       # 人物设定
      relationships.md             # 人物关系图谱
      outline.md                   # 故事大纲
      story.md                     # 最终故事
      review.md                    # 审核报告
```

## 4. D.E.S.I.R.E. Matrix 理论框架

### 4.1 六维分析矩阵

| 维度 | 全称 | 调用理论 | 历史故事适配 |
|------|------|----------|-------------|
| D | Drive 驱动力 | 自我决定论、五大性格特质、目标设定理论 | 这个事件为何流传千年？触动了哪种核心心理需求？ |
| E | Experience 体验角色 | 巴图玩家分类法、戏剧角色理论 | 读者渴望体验什么角色？ |
| S | Shortcut 认知捷径 | 行为经济学、说服心理学、认知偏误清单 | 读者可能存在哪些认知偏误？ |
| I | Identity 身份认同 | 荣格12原型理论、社会认同理论 | 触动了哪种集体身份认同？ |
| R | Risk 风险偏好 | 前景理论、调节定向理论、依恋理论 | 读者最怕什么？需规避什么误区？ |
| E² | Emotion 情感偏好 | Moral Foundations Theory、普拉切克情感轮、冯内古特弧线 | 最动人的情感弧线是什么？ |

### 4.2 WritingStyle 推导逻辑

Matrix 六维度到文风的映射链：

| Matrix 维度 | 文风影响 | 映射规则 |
|-------------|----------|----------|
| D - Drive | 创作理念 | 胜任感→展现智识；归属感→探索情感；自主→展现抗争 |
| E - Experience | 叙事节奏 | 成就者→快节奏行动；探索者→慢节奏描写；社交家→对话驱动 |
| S - Shortcut | 复杂度 | 容忍模糊→复杂情节；反之→简单直接 |
| I - Identity | 语言风格 | 智者→思辨典雅；英雄→激昂有力；颠覆者→直接冲击力 |
| R - Risk | 描写精细度 | 害怕模糊→微观精细描写；害怕无关→强代入感 |
| E² - Emotion | 核心基调 | 读者的"情感终点"=文风的最终氛围 |

### 4.3 剧情建议推导逻辑

| 剧情建议模块 | Matrix 来源 | 推导规则 |
|-------------|-------------|----------|
| 规避区 | R - Risk | 列出读者的"雷点"，剧情红线 |
| 核心驱动力 | D - Drive | 故事最根本的内在驱动力 |
| 推荐剧情方向 | D + E | 基于驱动力和体验角色的剧情脉络 |
| 可设计的剧情起伏 | E + I | 命中读者理想角色的核心矛盾 |
| 剧情奖励 | D | 满足读者核心驱动力的"奖励"情节 |

## 6. 非功能需求

### 6.1 性能
- 同一 Phase 内的多个事件必须并行处理
- 评估卡片必须紧凑（~200字/张），确保 Selector 步骤不会上下文爆炸

### 6.2 可靠性
- API 不可用时，WebSearch 作为降级方案
- 史实审核作为最后一道防线，重大错误返回修改

### 6.3 可扩展性
- 新增数据源只需修改 `fetch_history.py`
- 新增子代理只需创建 `agents/{role}.md` 并更新 SKILL.md
- 相邻 Phase 可合并到一个子代理执行以减少启动开销

## 8. 已知限制与优化方向

| 限制 | 优先级 | 说明 |
|------|--------|------|
| 评估卡片评分趋同 | P2 | 多个事件可能都评⭐5，需增加评分校准指导 |
| Phase 合并执行 | P2 | 实际运行中相邻 Phase 常合并到一个子代理，SKILL.md 应明确允许 |
| API SSL 问题 | P3 | Day in History API 存在 SSL 错误，需增加更多备用数据源 |
| 故事长度控制 | P3 | 自适应长度缺乏精确的字数约束机制 |
