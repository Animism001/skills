## 技能创建目录模板

标准目录Claude 的 Skill 标准目录结构是一个基于文件夹的模块化系统，核心设计理念是"渐进式披露"，旨在高效利用上下文窗口并按需加载资源。一个规范的 Skill 就是一个独立的文件夹，其标准目录结构如下：

```
[skill-name]/                  # 根目录：技能名称（必须使用小写字母、数字和连字符，如 code-reviewer）
├─ SKILL.md                    # 【必需】核心定义文件：包含 YAML 元数据+ Markdown 指令
├─ README.md                   # 【可选】给人类开发者看的说明文档
├─ LICENSE.txt                 # 【可选】许可证文件
├─ examples.md                 # 【可选】Few-Shot 示例：提供输入/输出对，规范 AI 行为
├─ scripts/                    # 【可选】工具脚本：Python/Bash/Node.js 等可执行脚本
│  └─ analyzer.py
├─ references/                 # 【可选】参考资料库：长文档、规范、Schema 等（按需加载）
│  ├─ style-guide.md
│  └─ api-spec.json
├─ assets/                     # 【可选】静态资源：模板、图片、字体、配置文件等
│  └─ template.xlsx
└─ templates/                  # 【可选】输出模板：固定格式的文本/代码模板
   └─ report-template.txt
```

### 核心解析

1. **核心大脑：SKILL.md** — Skill 的灵魂与唯一必需的文件。YAML 元数据（name、description）+ Markdown 指令体。Claude 启动时仅加载元数据（约100 Tokens），匹配后才加载正文。

2. **确定性执行引擎：scripts/** — 存放可执行脚本，弥补大模型在数学计算、API调用等需要确定性结果的任务中的幻觉问题。脚本代码不进入上下文，只有输入输出占用上下文。

3. **无限知识外脑：references/** — 存放长文档、API规范等，按需加载，彻底解决上下文溢出问题。

4. **格式保障：assets/ 与 templates/** — 静态资源与输出模板，确保输出格式统一。

5. **目录设计的精髓：渐进式披露** — 三层加载机制：
   - 第一层（始终加载）：仅读取 SKILL.md 中的 name 和 description
   - 第二层（触发加载）：匹配任务后，读取 SKILL.md 的正文指令
   - 第三层（按需加载）：只有在指令要求时，才读取 scripts/、references/ 等附属文件

## Sub-agent

在 Claude 的技能生态中，Sub-agent 是预配置的、拥有特定"人格"和"技能"的 AI 助手。当 Skill 目录中包含 agents/ 时，该技能就从单纯的"知识+脚本"升级为了"多智能体调度系统"。

### 为什么 Skill 需要 Sub-agent？

面对复杂的长程任务，单一的通用模型往往面临上下文污染和专业性不足的问题。通过在 Skill 中内置专门的 Sub-agent，可以让每个子代理在自己独立的上下文窗口中运行，互不干扰。

### Sub-agent 的配置规范

每个子代理通常是一个带有 YAML 头信息的 Markdown 文件：

```yaml
---
name: skill-creator-architect
description: Use PROACTIVELY when need to design skill architecture
tools: Read, Write, Bash
---
```

### 协同编排模式

- **顺序链**：先让"需求分析 Agent"梳理意图，再传给"架构设计 Agent"生成目录，最后由"校验 Agent"检查合规性
- **条件路由**：根据前序 Agent 的输出，决定后续路径
- **并行处理**：将独立子任务分发给多个 Agent 同时执行

## 多 Skill 组合成一个 Skill 的目录结构

将多个 Skill 组合成一个复合型 Skill，主要有三种典型范式：

### 一、依赖声明聚合型

通过 `depends_on` 字段声明依赖关系，将多个独立 Skill 连接成有向无环图（DAG）。各子 Skill 保持物理目录独立，父级 Skill 作为编排器。

### 二、子代理编排型

将多个具有特定"人格"与"技能"的 Sub-agent 封装到一个 Skill 的 agents/ 目录中，升级为"多智能体调度系统"。

### 三、工作流粘合型

在单个 Skill 目录下，通过丰富的子目录和 Markdown 链接进行内部粘合，形成自包含的"重型 Skill"。依靠渐进式披露机制在 SKILL.md 中做精准的路由与引用。

### 工程建议

- 弱依赖，重数据流转 → 依赖声明聚合型
- 强推理，需上下文隔离 → 子代理编排型
- 强耦合，重按需调度 → 工作流粘合型

核心原则：将领域知识装进组织良好的文件夹，让脚本替代大段自然语言描述，用确定性执行弥补模型幻觉，并让模型只看它该看的内容。
